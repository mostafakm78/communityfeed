# Community Feed

این پروژه با [Next.js](https://nextjs.org) ساخته شده است.

## شروع به کار

ابتدا سرور توسعه را اجرا کنید:

```bash
npm run dev
```

سپس آدرس [http://localhost:3000](http://localhost:3000) را در مرورگر باز کنید.

## ساختار پروژه

برای این پروژه، بر اساس نیازهای پروژه، معماری Feature-Based انتخاب شده است؛ به این صورت که هر فیچر به‌طور مستقل از بقیه جدا شده و تمام وابستگی‌های آن (کامپوننت‌ها، توابع دریافت داده، تایپ‌ها و...) در کنار هم در یک پوشه مشترک قرار می‌گیرند. این رویکرد در توسعه‌های آینده، پیدا کردن و توسعه‌دادن هر بخش از پروژه را ساده‌تر می‌کند.

## تصمیمات معماری

برای صفحه لیست سوالات، از آن‌جا که این بخش دارای اسکرول بی‌نهایت (Infinite Scroll) است، از رویکرد ISR (Incremental Static Regeneration) استفاده شده تا داده سوالات در بازه‌های زمانی مشخص از سرور به‌روزرسانی شود، بدون این‌که با هر درخواست کاربر، فشار اضافی روی سرور وارد شود.

برای صفحه تک سوال و پاسخ‌های آن، به‌طور کامل از Server-Side Rendering استفاده شده است تا صفحه با سریع‌ترین حالت ممکن برای کاربر ساخته شود؛ و از آن‌جا که این صفحه یک صفحه جزئیات (Detail Page) است، این معماری برای SEO نیز مناسب‌تر است.

### مدیریت State

داده‌های این پروژه بر اساس ماهیت‌شان، نه بر اساس یک ابزار واحد، در سه لایه متفاوت نگه‌داری می‌شوند:

- **URL State:** پارامترهای جستجو، فیلتر (`book`) و مرتب‌سازی (`sort`) در Query String نگه‌داری می‌شوند تا لینک هر حالت از صفحه قابل اشتراک‌گذاری باشد و با رفرش‌شدن صفحه از بین نرود.
- **Server Cache State:** لیست سوالات، جزئیات سوال و پاسخ‌ها در هیچ Store دستی نگه‌داری نمی‌شوند؛ کش‌شدن، Revalidate و Invalidate این داده‌ها به‌طور کامل به React Query سپرده شده است.
- **Local State:** مقادیر لحظه‌ای فیلدهای فرم، باز/بسته‌بودن Dropdown‌ها و Modal‌ها، و وضعیت در‌حال‌درگ‌بودن فایل، همگی با `useState` در همان کامپوننت مربوطه نگه‌داری می‌شوند و به بیرون از آن کامپوننت درز نمی‌کنند.

برای این MVP، عمداً هیچ کتابخانه State سراسری (مثل Zustand یا Redux) معرفی نشده است؛ چون با بررسی دقیق داده‌های پروژه، هیچ موردی پیدا نشد که واقعاً به‌طور هم‌زمان توسط چند بخش غیرمرتبط از اپلیکیشن نیاز باشد. هر داده در لایه‌ای قرار گرفته که بیشترین تطابق را با ماهیتش دارد، نه در ابزاری که از پیش انتخاب شده باشد.

### فرضیات و تصمیمات تکمیلی نسبت به Brief

در بررسی داکیومنت Swagger، چند مورد اختلاف بین توضیحات Brief و Schema واقعی API مشاهده شد. تصمیمات زیر برای رفع این موارد گرفته شده‌اند:

- **تگ → فیلد book:** Brief از یک سیستم تگ عمومی با محدودیت حداکثر ۵ مورد صحبت می‌کند، اما Schema واقعی API نشان می‌دهد دسته‌بندی سوالات از طریق فیلدهای ساختاریافته (`study_field`، `education_level`، `education_grade`، `book`) انجام می‌شود، نه یک سیستم تگ آزاد. در فرم ثبت سوال، از بین این فیلدها فقط `book` به‌عنوان جایگزین تگ در نظر گرفته شد، چون این فیلد در API به‌صورت یک رشته تکی تعریف شده و هر سوال منطقاً به یک کتاب درسی مشخص تعلق دارد؛ به همین دلیل به‌جای ورودی چندمقداری، یک Select تک‌انتخابی و اجباری برای `book` پیاده‌سازی و قانون «حداکثر ۵ تگ» کنار گذاشته شد.
- **عنوان و توضیحات → فیلد واحد content:** Brief عنوان و توضیحات را دو فیلد جدا با اعتبارسنجی مستقل تعریف کرده، اما API واقعی فقط یک فیلد به‌نام `content` دارد و فیلدی مجزا برای عنوان وجود ندارد. برای حفظ تجربه کاربری مدنظر Brief، در فرم همچنان دو فیلد عنوان و توضیحات جدا از هم نمایش داده می‌شود (با همان قواعد اعتبارسنجی)، اما پیش از ارسال، این دو مقدار در یک رشته `content` واحد ترکیب می‌شوند.
- **احراز هویت:** نه در Brief و نه در Swagger اشاره‌ای به Authentication نشده، بنابراین این بخش به‌طور کامل خارج از محدوده این MVP در نظر گرفته شده است. ساختار Axios Client (در `lib/axios.ts`) به‌گونه‌ای طراحی شده که در صورت نیاز آینده، افزودن Token از طریق Interceptor بدون نیاز به تغییر در لایه سرویس‌ها امکان‌پذیر باشد.
- **سقف کاراکتر توضیحات:** Brief فقط حداقل طول توضیحات (۵۰ کاراکتر) را مشخص کرده و سقفی برای آن ذکر نکرده است. برای پیشگیری از ارسال محتوای بیش‌از‌حد طولانی، سقف ۳۰۰۰ کاراکتر برای این فیلد در نظر گرفته شد؛ عددی که برای یک سوال فنی همراه با توضیح کامل مسئله و مراحل بررسی‌شده کافی است، بدون آن‌که ارسال محتوای حجیم در این مرحله از MVP را تشویق کند.

## دلایل انتخاب روش‌های دریافت داده

- **لیست سوالات (صفحه اصلی):** صفحه اول لیست با `unstable_cache` و `revalidate: 600` (هر ۱۰ دقیقه) روی سرور کش می‌شود و از طریق `HydrationBoundary` به React Query کلاینت منتقل (Hydrate) می‌شود. به این ترتیب اولین بار که کاربر صفحه را باز می‌کند، نیازی به یک درخواست تکراری از کلاینت برای صفحه اول نیست. از طرفی چون اسکرول بی‌نهایت است و صفحات بعدی به‌صورت پویا و بسته به موقعیت اسکرول کاربر درخواست می‌شوند، نمی‌توان آن‌ها را از پیش کش یا ISR کرد؛ به همین دلیل واکشی صفحات بعدی با `useInfiniteQuery` به‌طور کامل روی کلاینت انجام می‌شود.
- **جزئیات سوال و پاسخ‌ها:** برای جزئیات سوال از تابع `cache()` خود React استفاده شده، نه `unstable_cache`؛ چون این داده باید همیشه تازه (Fresh) باشد (مثلاً وقتی پاسخ جدیدی ثبت می‌شود) و فقط نیاز است در طول یک رندر سمت سرور، فراخوانی‌های تکراری (در `page.tsx`، `QuestionDetails` و `QuestionsRelated`) به یک درخواست واحد تجمیع شوند، بدون اینکه بین درخواست‌های مختلف کاربران کش بماند.
- **جستجو:** واکشی نتایج جستجو به‌طور کامل سمت کلاینت و با `AbortController` انجام می‌شود تا با هر کلیدی که کاربر تایپ می‌کند، درخواست قبلی لغو شده و نتیجه درخواست‌های قدیمی، نتیجه جدیدتر را بازنویسی نکند (جلوگیری از Race Condition). این بخش تعاملی و وابسته به ورودی لحظه‌ای کاربر است، پس SSR برایش سودی ندارد.
- **ثبت سوال جدید:** ارسال فرم با `useMutation` از React Query انجام می‌شود چون این یک عملیات نوشتن (Write) است، نه خواندن؛ پس مفهوم کش یا Revalidate برایش معنا ندارد و فقط مدیریت وضعیت ارسال (Pending/Success/Error) با Toast و ریدایرکت اهمیت دارد.
- **لیست کتاب‌ها در فرم ثبت سوال:** با `useQuery` فقط در سمت کلاینت و در زمان نیاز (هنگام رندر شدن فیلد انتخاب کتاب) واکشی می‌شود؛ چون این لیست در نتایج موتورهای جستجو اهمیتی ندارد و واکشی آن می‌تواند بدون نیاز به SSR، به‌محض نمایش فرم انجام شود.

## بهینه‌سازی‌های عملکردی اعمال‌شده

تمام موارد زیر با `next/dynamic` و بدون غیرفعال‌کردن SSR (یعنی بدون `ssr: false`) انجام شده‌اند؛ هدف فقط کوچک‌تر کردن باندل جاوااسکریپت اولیه و موازی‌سازی دانلود چانک‌هاست، بدون این‌که محتوای اولیه صفحه (و در نتیجه SEO و LCP) قربانی شود:

- **صفحه اصلی / لیست سوالات:** `QuestionsListClient` (که شامل منطق `useInfiniteQuery` و `IntersectionObserver` برای اسکرول بی‌نهایت است) با `next/dynamic` از داخل کامپوننت سرور `QuestionsList` بارگذاری می‌شود تا این منطق در یک چانک جدا از باندل اصلی صفحه قرار گیرد. در ادامه، `QuestionCard` نیز داخل `QuestionsListClient` به‌صورت dynamic و با یک Fallback اسکلت (`QuestionCardSkeleton`) بارگذاری می‌شود. در مقابل، کامپوننت‌های کوچک مثل `AnswerCount`، `QuestionInfo`، `Tags` و `QuestionsListHeader` عمداً lazy نشده‌اند چون آن‌قدر کوچک هستند که سربار شبکه و Suspense اضافه، بیشتر از صرفه‌جویی واقعی است.
- **صفحه جزئیات سوال:** کل این بخش (از `QuestionDetails` و `QuestionsRelated` تا `QuestionCard`، `AnswerList`، `AnswerCard` و...) کامپوننت سرور است و هیچ جزء کلاینتی (به‌جز یک Separator بسیار کوچک) ندارد؛ بنابراین Dynamic Import در این بخش سودی ندارد. در عوض، با دو Suspense Boundary مستقل برای `QuestionsRelated` و `QuestionDetails` در `page.tsx`، این دو بخش به‌صورت موازی استریم می‌شوند و هیچ‌کدام منتظر دیگری نمی‌ماند.
- **صفحه ثبت سوال جدید:** `NewAnswerForm` (که شامل `react-hook-form`، `zod`، `@hookform/resolvers` و `useMutation` از React Query است) با `next/dynamic` از کامپوننت سرور صفحه import می‌شود تا این باندل سنگین در یک چانک جدا قرار گیرد. داخل خود فرم نیز، `SelectBook` (که شامل کامپوننت Select از Radix و یک `useQuery` برای دریافت لیست کتاب‌هاست) با `next/dynamic` و یک Fallback از نوع `Skeleton` (هم‌اندازه با `SelectTrigger`) جداگانه بارگذاری می‌شود.
- **کش سمت سرور:** استفاده از `unstable_cache` با `revalidate: 600` برای اولین صفحه لیست سوالات، تعداد درخواست‌های مستقیم به API بک‌اند را در بازه‌های ۱۰ دقیقه‌ای محدود می‌کند. همچنین استفاده از `cache()` در `getQuestionDetail` باعث می‌شود سه فراخوانی هم‌نام این تابع در یک رندر سمت سرور (در `page.tsx`، `QuestionDetails` و `QuestionsRelated`) به یک درخواست واحد به API تجمیع شوند.

## اگر یک روز زمان بیشتر داشتم

- **داشبورد سوالات من:** پیاده‌سازی یک صفحه داشبورد شخصی مبتنی بر Endpoint مربوط به سوالات کاربر، جایی که بتواند سوالات خود را در یک‌جا ببیند و حالت‌های مختلف مدیریت آن‌ها (از جمله ویرایش و حذف، همراه با Confirm Dialog پیش از حذف و بازخورد مناسب بعد از هر عملیات) را انجام دهد.
- **بهبود انتخاب کتاب:** فعلاً لیست کتاب‌ها به‌صورت ثابت در نظر گرفته شده است. در صورت داشتن زمان بیشتر، در نظر داشتم این لیست را به‌جای یک Select ساده، به‌صورت داینامیک و وابسته به فیلدهای مرتبط (مثل رشته یا پایه تحصیلی) پیاده‌سازی کنم تا انتخاب کتاب دقیق‌تر و کاربرپسندتر شود.
- **Draft Recovery:** ذخیره خودکار محتوای فرم ثبت سوال در `localStorage` (با Debounce روی تغییرات)، به‌گونه‌ای که اگر کاربر فرم را ناتمام رها کند و بعداً به صفحه برگردد، متن واردشده از بین نرفته باشد.
